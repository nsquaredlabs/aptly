## [v0.3.0] - 2026-01-27

### Added
- **Response PII Detection** - Now scans LLM responses for PII before returning to clients
  - New `redact_response` parameter (default: `false`) to enable response scanning
  - Response PII detections tracked separately in audit logs (`response_pii` field)
  - Same entity types and confidence thresholds as input detection
  - See [Response PII Detection PR](https://github.com/your-org/aptly/pull/XXX) for details

- **Production Deployment Infrastructure**
  - Sentry integration for error tracking and performance monitoring
  - Automated deployment verification scripts (`scripts/smoke_test.py`, `scripts/verify_deployment.sh`)
  - Complete production deployment guide (DEPLOYMENT.md)
  - Railway deployment configuration (`railway.toml`, `Procfile`)

### Changed
- Migration `002_response_pii.sql` adds `response_pii` JSONB column to `audit_logs` table
- Audit log response format now includes `response_pii_detections` field
- Improved error handling for LLM provider failures

### Fixed
- Fixed spaCy model installation for Presidio (use `en_core_web_sm` instead of `en_core_web_lg`)
- Fixed Railway port binding (use dynamic `$PORT` environment variable)

### Documentation
- Added deployment guide with Railway setup instructions
- Added troubleshooting section for common deployment issues
- Updated API reference with response PII detection examples

## [v0.1.0] - 2026-01-25

### Added - Initial MVP Release

#### Core Features
- **PII Redaction** - Automatic detection and redaction of sensitive data
  - Powered by Microsoft Presidio
  - Three redaction modes: `mask` (default), `hash`, `remove`
  - Supports 15+ entity types: PERSON, EMAIL, SSN, CREDIT_CARD, etc.
  - Configurable per customer via `pii_redaction_mode` setting

- **Chat Completions API** - OpenAI-compatible endpoint
  - POST `/v1/chat/completions` endpoint
  - Multi-provider support via LiteLLM (OpenAI, Anthropic, Google, Cohere, Together)
  - Customer-provided API keys (pass-through, never stored)
  - Streaming and non-streaming responses

- **Audit Logging** - Immutable request/response logs
  - Every request logged with full metadata
  - Database trigger prevents modification/deletion
  - Queryable via GET `/v1/logs` with pagination
  - Includes: timestamps, models, costs, token usage, PII detections

- **Authentication System**
  - Bootstrap authentication via `APTLY_ADMIN_SECRET` environment variable
  - Admin endpoints use `X-Admin-Secret` header
  - Customer endpoints use `Authorization: Bearer apt_live_*` API keys
  - API keys stored as bcrypt hashes
  - Support for test (`apt_test_*`) and live (`apt_live_*`) keys

- **Rate Limiting** - Redis-based distributed rate limiting
  - Per-customer rate limits based on plan (starter: 100/hr, pro: 1000/hr)
  - Fail-open behavior when Redis unavailable
  - Rate limit headers in responses (`X-RateLimit-*`)

#### API Endpoints

**Admin Endpoints** (X-Admin-Secret authentication):
- `POST /v1/admin/customers` - Create customer account with API key
- `GET /v1/admin/customers` - List all customers
- `GET /v1/admin/customers/{id}` - Get customer details
- `POST /v1/admin/customers/{id}/api-keys` - Create API key for customer

**Customer Endpoints** (Bearer token authentication):
- `POST /v1/chat/completions` - Chat completion with PII redaction
- `GET /v1/api-keys` - List your API keys
- `POST /v1/api-keys` - Create new API key
- `DELETE /v1/api-keys/{id}` - Revoke API key
- `GET /v1/logs` - Query audit logs with pagination
- `GET /v1/logs/{id}` - Get single audit log detail
- `GET /v1/me` - Get customer profile
- `PATCH /v1/me` - Update customer settings

**Public Endpoints**:
- `GET /v1/health` - Health check (database + Redis connectivity)

#### Database Schema
- `customers` table - Customer accounts with plan and compliance settings
- `api_keys` table - API keys with bcrypt hashing
- `audit_logs` table - Immutable request logs (protected by database trigger)

#### Tech Stack
- **FastAPI** - Web framework
- **Supabase** - PostgreSQL database
- **Redis** - Distributed rate limiting
- **LiteLLM** - Multi-provider LLM abstraction
- **Microsoft Presidio** - PII detection engine
- **spaCy** - NLP model for Presidio

#### Compliance Features
- HIPAA-compliant audit logging
- PII never sent to external LLMs
- Immutable audit trail with database enforcement
- Customer-controlled LLM API keys (never stored)

### Documentation
- README.md with quickstart guide
- SPEC.md product specification
- DEPLOYMENT.md production deployment guide
- CLAUDE.md AI assistant guidance
- 95 passing tests with examples

### Known Limitations
- No web dashboard (API only)
- No webhook notifications
- No custom PII entity types
- No response caching
- Single region deployment only

## Migration Guides

### Upgrading to v0.3.0 from v0.2.x

**Database migration required:**

```bash
# Run migration 002_response_pii.sql
supabase db push

# Or manually execute:
ALTER TABLE audit_logs ADD COLUMN response_pii JSONB DEFAULT '{"detected": false}'::jsonb;
```

**API changes:**
- Response format now includes `response_pii_detections` field (will be `null` unless `redact_response: true`)
- No breaking changes to existing endpoints

**Action required:**
- Run database migration before deploying v0.3.0
- Update clients if you want to use response PII detection

### Upgrading to v0.2.0 from v0.1.x

**No database migration required.**

**API changes:**
- Added `stream` parameter to chat completions
- No breaking changes to existing endpoints

**Action required:**
- None (fully backward compatible)
- Optionally update clients to use streaming
