# Troubleshooting Guide

This guide covers common issues you may encounter when using Aptly and how to resolve them.

## Authentication Issues

### 401 Unauthorized

**Symptom:** API requests fail with `401 Unauthorized` error.

**Common Causes:**

<AccordionGroup>
  <Accordion title="Missing or Invalid API Key">
    **Solution:**
    - Verify your API key starts with `apt_live_` (production) or `apt_test_` (test mode)
    - Ensure you're using the `Authorization: Bearer` header format
    - Check for extra spaces or newline characters in the key

    ```bash
    # Correct format
    curl https://api.aptly.dev/v1/logs \
      -H "Authorization: Bearer apt_live_xyz789..."

    # Common mistake - missing "Bearer"
    curl https://api.aptly.dev/v1/logs \
      -H "Authorization: apt_live_xyz789..."  # ❌ Wrong
    ```
  </Accordion>

  <Accordion title="Revoked API Key">
    **Solution:**
    - Check if the API key was revoked using `GET /v1/api-keys`
    - Create a new API key if needed: `POST /v1/api-keys`
    - Update your application with the new key
  </Accordion>

  <Accordion title="Wrong Environment">
    **Solution:**
    - Test keys (`apt_test_*`) only work in test mode
    - Live keys (`apt_live_*`) are for production
    - Verify you're using the correct key for your environment
  </Accordion>
</AccordionGroup>

### 403 Forbidden

**Symptom:** API request is authenticated but denied access.

**Cause:** Your API key doesn't have permission to access the requested resource (e.g., trying to access another customer's data).

**Solution:**
- Verify you're accessing resources that belong to your customer account
- Check that you're using a customer API key, not an admin secret
- For admin operations, use the `X-Admin-Secret` header instead

## LLM Provider Issues

### 402 Payment Required

**Symptom:** Chat completion requests fail with `402 Payment Required`.

**Common Causes:**

<AccordionGroup>
  <Accordion title="Invalid LLM Provider API Key">
    **Solution:**
    - Verify the provider API key is valid and active
    - Check that the key has sufficient credits/quota
    - Ensure the key matches the model provider:
      - OpenAI models (`gpt-4`, `gpt-3.5-turbo`) need OpenAI keys
      - Anthropic models (`claude-*`) need Anthropic keys
      - Google models (`gemini-*`) need Google AI keys

    ```python
    # Correct
    response = requests.post(
        "https://api.aptly.dev/v1/chat/completions",
        json={
            "model": "gpt-4",
            "api_keys": {"openai": "sk-..."}  # ✅ Correct provider
        }
    )

    # Wrong
    response = requests.post(
        "https://api.aptly.dev/v1/chat/completions",
        json={
            "model": "gpt-4",
            "api_keys": {"anthropic": "sk-ant-..."}  # ❌ Wrong provider
        }
    )
    ```
  </Accordion>

  <Accordion title="Expired or Depleted Credits">
    **Solution:**
    - Log into your LLM provider dashboard (OpenAI, Anthropic, etc.)
    - Check your billing status and add credits if needed
    - Verify your API key hasn't expired
  </Accordion>
</AccordionGroup>

### 500 Bad Gateway / LLM Provider Error

**Symptom:** Requests fail with `500` or `502` errors referencing the LLM provider.

**Solutions:**

1. **Check Provider Status:**
   - Visit provider status pages:
     - OpenAI: [status.openai.com](https://status.openai.com)
     - Anthropic: [status.anthropic.com](https://status.anthropic.com)
   - Wait and retry if there's an outage

2. **Verify Model Name:**
   ```python
   # Correct model names
   "gpt-4"
   "gpt-3.5-turbo"
   "claude-3-5-sonnet-20241022"
   "gemini-1.5-pro"

   # Common mistakes
   "gpt4"           # ❌ Missing hyphen
   "claude-3.5"     # ❌ Incomplete version
   ```

3. **Check Request Payload:**
   - Ensure messages array is not empty
   - Verify message roles are valid (`user`, `assistant`, `system`)
   - Check that content is a string, not null

## Rate Limiting

### 429 Too Many Requests

**Symptom:** Requests fail with `429 Too Many Requests` error.

**Understanding Rate Limits:**

Aptly enforces rate limits based on your plan. Check the response headers:

```bash
X-RateLimit-Limit: 1000        # Max requests per hour
X-RateLimit-Remaining: 0       # Requests left this hour
X-RateLimit-Reset: 1706616000  # Unix timestamp when limit resets
```

**Solutions:**

<AccordionGroup>
  <Accordion title="Implement Retry Logic with Backoff">
    ```python
    import time
    import requests
    from requests.adapters import HTTPAdapter
    from urllib3.util.retry import Retry

    # Configure automatic retries with exponential backoff
    session = requests.Session()
    retries = Retry(
        total=3,
        backoff_factor=1,
        status_forcelist=[429, 500, 502, 503, 504],
        allowed_methods=["GET", "POST"]
    )
    session.mount('https://', HTTPAdapter(max_retries=retries))

    response = session.post(
        "https://api.aptly.dev/v1/chat/completions",
        headers={"Authorization": f"Bearer {api_key}"},
        json={...}
    )
    ```
  </Accordion>

  <Accordion title="Check Usage and Upgrade Plan">
    **Monitor your usage:**
    ```bash
    curl https://api.aptly.dev/v1/me \
      -H "Authorization: Bearer $APTLY_API_KEY"
    ```

    **Contact support** if you need higher rate limits:
    - Email: support@aptly.dev
    - Include your customer ID and use case
  </Accordion>

  <Accordion title="Batch Requests Efficiently">
    - Avoid making requests in tight loops
    - Use async/concurrent requests with proper rate limiting
    - Cache responses when possible
    - Consider using streaming for long conversations
  </Accordion>
</AccordionGroup>

## PII Detection Issues

### PII Not Being Detected

**Symptom:** Expected PII is not showing up in `pii_detections` field.

**Common Causes:**

<AccordionGroup>
  <Accordion title="Low Confidence Score">
    **Explanation:** Presidio only returns detections above a confidence threshold (default: 0.5).

    **Solution:** Check the entity type and text format:

    ```python
    # High confidence (usually detected)
    "My SSN is 123-45-6789"        # ✅ Standard format
    "Email: john@example.com"      # ✅ Clear email
    "Card: 4111-1111-1111-1111"    # ✅ Standard card format

    # Lower confidence (might be missed)
    "My SSN is 123 45 6789"        # ⚠️ Non-standard spacing
    "Contact me at john at example" # ⚠️ Obfuscated email
    ```
  </Accordion>

  <Accordion title="Unsupported Entity Type">
    **Supported entity types:**
    - `PERSON` - Names
    - `EMAIL_ADDRESS` - Email addresses
    - `PHONE_NUMBER` - Phone numbers
    - `US_SSN` - Social Security Numbers
    - `CREDIT_CARD` - Credit card numbers
    - `US_PASSPORT` - Passport numbers
    - `LOCATION` - Addresses, cities
    - `DATE_TIME` - Dates and times
    - `MEDICAL_LICENSE` - Medical license numbers
    - `US_DRIVER_LICENSE` - Driver's license numbers
    - `IBAN_CODE` - International bank account numbers

    **Check the [PII Redaction Guide](/guides/pii-redaction) for the complete list.**
  </Accordion>

  <Accordion title="Context Matters">
    Some PII requires context to be detected:

    ```python
    # More likely to be detected
    "Patient name: John Smith"
    "SSN: 123-45-6789"

    # Might be missed without context
    "John Smith"  # Could be anyone
    "123456789"   # Could be any number
    ```
  </Accordion>
</AccordionGroup>

### False Positives

**Symptom:** Non-PII text is being redacted.

**Solution:**

This is expected behavior for high-confidence detection. Presidio errs on the side of caution for compliance.

**Workarounds:**
1. **Use hash mode** for consistent redaction: `"pii_redaction_mode": "hash"`
2. **Disable response scanning** if only input needs redaction: `"redact_response": false`
3. **Review detections** and adjust your prompts to avoid ambiguous text

## Streaming Issues

### Streaming Response Cuts Off

**Symptom:** Streaming response stops mid-sentence or doesn't complete.

**Solutions:**

<AccordionGroup>
  <Accordion title="Client Timeout">
    Increase your HTTP client timeout:

    ```python
    response = requests.post(
        "https://api.aptly.dev/v1/chat/completions",
        json={"stream": True, ...},
        stream=True,
        timeout=300  # 5 minutes
    )
    ```
  </Accordion>

  <Accordion title="Connection Dropped">
    Implement reconnection logic:

    ```python
    def stream_with_retry(url, headers, data, max_retries=3):
        for attempt in range(max_retries):
            try:
                response = requests.post(
                    url, headers=headers, json=data,
                    stream=True, timeout=300
                )
                for line in response.iter_lines():
                    if line:
                        yield line
                break  # Success
            except requests.exceptions.RequestException as e:
                if attempt == max_retries - 1:
                    raise
                time.sleep(2 ** attempt)  # Exponential backoff
    ```
  </Accordion>

  <Accordion title="Check for [DONE] Event">
    Always check for the final event:

    ```python
    for line in response.iter_lines():
        if line:
            line_str = line.decode('utf-8')
            if line_str.startswith('data: '):
                data = line_str[6:]  # Remove 'data: ' prefix
                if data == '[DONE]':
                    break  # Stream complete
                chunk = json.loads(data)
                # Process chunk...
    ```
  </Accordion>
</AccordionGroup>

### No Data in Stream

**Symptom:** Connection succeeds but no chunks are received.

**Solutions:**

1. **Verify `stream: true` is set:**
   ```python
   json={"stream": True, ...}  # Not "stream": "true" (string)
   ```

2. **Check model supports streaming:**
   - All major models support streaming (GPT-4, Claude, Gemini)
   - Verify model name is correct

3. **Ensure proper iteration:**
   ```python
   # Correct
   for line in response.iter_lines():
       if line:
           print(line.decode('utf-8'))

   # Wrong - will block forever
   data = response.json()  # ❌ Don't use .json() with streaming
   ```

## Audit Log Issues

### Logs Missing or Incomplete

**Symptom:** Expected audit logs don't appear or are incomplete.

**Solutions:**

<AccordionGroup>
  <Accordion title="Check Time Range">
    Logs are paginated by default. Use `start_time` and `end_time`:

    ```bash
    curl "https://api.aptly.dev/v1/logs?start_time=2026-01-01T00:00:00Z&end_time=2026-01-31T23:59:59Z" \
      -H "Authorization: Bearer $APTLY_API_KEY"
    ```
  </Accordion>

  <Accordion title="Request Failed Before Logging">
    Logs are only created for requests that reach the LLM provider. If authentication or validation fails early, no log is created.

    **Check for:**
    - 401/403 errors (authentication failed)
    - 400 errors (validation failed)
    - These are logged separately in error logs
  </Accordion>

  <Accordion title="Pagination">
    Use pagination parameters to retrieve all logs:

    ```python
    all_logs = []
    offset = 0
    limit = 100

    while True:
        response = requests.get(
            f"https://api.aptly.dev/v1/logs?offset={offset}&limit={limit}",
            headers={"Authorization": f"Bearer {api_key}"}
        )
        data = response.json()
        all_logs.extend(data['logs'])

        if len(data['logs']) < limit:
            break  # No more logs
        offset += limit
    ```
  </Accordion>
</AccordionGroup>

## Connection and Network Issues

### Connection Timeout

**Symptom:** Requests time out before completing.

**Solutions:**

1. **Increase timeout values:**
   ```python
   response = requests.post(
       "https://api.aptly.dev/v1/chat/completions",
       timeout=30  # Default is too low for LLM requests
   )
   ```

2. **Check network connectivity:**
   ```bash
   # Test basic connectivity
   curl https://api.aptly.dev/v1/health

   # Check DNS resolution
   nslookup api.aptly.dev

   # Test with verbose output
   curl -v https://api.aptly.dev/v1/health
   ```

3. **Firewall/proxy issues:**
   - Ensure outbound HTTPS (port 443) is allowed
   - Configure proxy settings if behind corporate firewall
   - Check for SSL/TLS certificate validation issues

### SSL Certificate Errors

**Symptom:** SSL certificate verification fails.

**Solutions:**

```python
# Verify your system's CA certificates are up to date
pip install --upgrade certifi

# If behind corporate proxy with custom CA
import requests
response = requests.post(
    "https://api.aptly.dev/v1/chat/completions",
    verify='/path/to/corporate-ca-bundle.crt'
)
```

## Performance Issues

### Slow Response Times

**Symptom:** Requests take longer than expected.

**Diagnosis:**

1. **Check response headers:**
   ```bash
   curl -w "\nTime: %{time_total}s\n" https://api.aptly.dev/v1/health
   ```

2. **Identify bottleneck:**
   - PII detection: ~50-200ms (depends on message length)
   - LLM provider: 1-30s (depends on model and response length)
   - Network latency: 10-500ms (depends on location)

**Solutions:**

<AccordionGroup>
  <Accordion title="Use Streaming for Long Responses">
    Streaming provides faster time-to-first-token:

    ```python
    response = requests.post(
        "https://api.aptly.dev/v1/chat/completions",
        json={"stream": True, ...},  # ✅ Faster perceived performance
        stream=True
    )
    ```
  </Accordion>

  <Accordion title="Optimize Message Length">
    - Keep messages concise when possible
    - PII detection time scales with text length
    - Use `max_tokens` to limit response length
  </Accordion>

  <Accordion title="Choose Faster Models">
    - GPT-3.5-turbo: Fastest, cheapest
    - GPT-4-turbo: Good balance
    - GPT-4: Slowest, highest quality
  </Accordion>
</AccordionGroup>
